從0到1的藥局採購管理POS系統

Overview

專案類型
Android | 從0到1的藥局採購管理POS系統
擔任腳色
UI / UX Designer
負責項目
負責UI畫面設計、元件規劃，以及各功能的操作流程設計

專案概述

傳統藥局訂貨仰賴電話與紙本傳真，流程繁瑣且無法即時掌握庫存與銷售數據，因此規劃一套整合訂貨、收銀與庫存管理的藥局 POS 系統

專案目標

打造一套適合地方小藥局使用的POS 系統，讓訂貨、收銀與庫存管理流程全面數位化，並降低日常操作門檻。

最終成果

完成一套藥局端可實際使用的輕量型 POS 系統，提升訂貨效率，並讓供應商即時掌握庫存與銷售數據。

使用者與應用場景User&Context

主要使用者：地方小型藥局（非科技背景）

使用情境：

  • 櫃檯收銀與日常結帳
  • 商品進出貨與庫存管理
  • 向供應商下單補貨
  • 查看營業與銷售報表

公司端需求：

  • 即時掌握藥局銷售量
  • 監控庫存／補貨行為
  • 蒐集各地數據做營運決策

專案挑戰Challenges

  • 從零經驗開始做 App UI/UX，跨領域學習曲線高
  • 對 Android 的 Material Design 規範完全不熟悉
  • 專案初期需求不明確，導致方向與功能取捨難度高

核心功能Core Features

庫存管理系統

  • 管理商品進貨、退貨、採購與盤點流程
  • 即時掌握庫存數量與商品狀態

商品進貨流程

商品採購及盤點

收銀與結帳系統

  • 提供快速、直覺的商品結帳操作流程
  • 支援商品搜尋、數量調整與金額計算

首頁可新增常用功能

購物車多店鋪結帳流程改版

Overview

專案類型
Web | 購物車多店鋪結帳流程改版
擔任腳色
UI / UX Designer
負責項目
主要負責結帳流程的規劃與前端畫面設計，包含購物車結構調整與訂單資訊呈現。

專案概述

原本的購物網站僅支援廠商與倉庫出貨，購物車結構相對單純。隨著未來規劃新增藥局端出貨，既有流程已無法因應多店鋪同時結帳，以及不同出貨來源在優惠、運費與退貨上的差異。

專案目標

在不影響原有使用流程的前提下，重新設計購物車與結帳體驗，讓系統能支援多店鋪、多優惠與多出貨來源，並保留後續擴充的彈性。

最終成果

完成一套以「店鋪為單位」的購物車與結帳流程設計，讓不同優惠、運費與訂單資訊能被清楚整理與呈現，同時維持既有廠商與倉庫出貨流程的順暢度。

設計方向Constraints

在這次改版中，重點是重新整理結帳流程的結構，讓系統在變得更複雜的同時，使用者仍然能清楚理解自己在做什麼。 整體設計方向包含：

  • 以「店鋪」作為購物車的主要結構單位
  • 將優惠、寄送方式與金額資訊集中在同一區塊中完成選擇
  • 確保未來即使加入新的出貨來源，也不會影響既有廠商與倉庫的操作流程

舊版購物車

專案挑戰Challenges

在這次購物車改版中，最大的挑戰不在畫面本身，而是如何在既有架構下，因應更複雜的出貨與優惠規則。

  • 流程需擴充，但不能影響既有體驗 新增藥局端出貨的同時，仍需維持原有廠商與倉庫出貨流程的順暢度。
  • 多店鋪、多規則的資訊整理 不同店鋪各自擁有優惠、運費與寄送方式，需讓使用者在結帳時清楚理解。
  • 兼顧當下需求與未來彈性 藥局端尚未上線，但流程設計需預留擴充空間，避免未來再次重構。

解決方案Solution

購物車流程改版

  • 原本的購物車只有清單，本次改版後，購物車改以「店鋪分區顯示」的方式呈現。
  • 每間店鋪可獨立完成商品、免運券、寄送方式與小計確認
  • 操作流程更直覺，不怕多店鋪、多規則搞混

結帳後體驗設計

  • 付款與退款狀態
  • 出貨進度
  • 分別的退貨方式

即使未來加入藥局端出貨，也能在相同結構下呈現，不需要再額外調整使用流程。
